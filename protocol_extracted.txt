1 Overview
The Biolight Patient Data Sharing Protocol (hereinafter referred to as the Biolight PDS
Protocol) is used to export the data of the Biolight products through the TCP/IP network. As the
server, the Biolight products can support multiple clients to connect to it. The Biolight PDS
protocol is compatible with part of the HL7 V2.6 protocol and provides the following two types of
data:
(1)
Unsolicited information. The data unsolicited sent include the online and offline status
of the module, the parameters supported by the module, the result values of the
parameters, physiological alarm information and technical alarm information. Once the
connection is established, the server periodically sends data to the client. The server
that sends this message is defined as an unsolicited sending interface.
(2)
Information requested by the client. Once the connection is established, the server will
wait for the client to send the requested data information, and the server will send the
requested specific data to the client after receiving the message.The server that sends
and receives this message has two interfaces: the real-time sending interface and the
query interface.
HL7 messages defined in the Paulette Patient Data Sharing Agreement include:

ORU_R01 message: module on/off message, module support parameter message,
periodic parameter message, aperiodic parameter message, physiological alarm message,
technical alarm message, parameter alarm limit message, parameter alarm level message,
parameter alarm switch message, waveform data message, heartbeat packet message, etc.
The above messages are distinguished by different message control ids (the MSH-10
field is the message control ID. For the definition of the control ID, see A.5 Control and
Settings).

QRY_R02 message: data query message, which can be classified according to the query
definition in the QRF_5 field. The query message types include physiological parameter
data, physiological alarm data, technical alarm data, waveform data, etc. For details, see
5.2 QRY^R02 Message.

ORF_R04 message: indicates the data query result message. The message is the
response to the QRY_R02 query message. For details, see 5.4 ORF^R04 Message.

QRY_A19 message: ADT patient query message. This message can be located based on
the medical record number in the QRD_8 field. For details, see 7.2 ADT Query Message
(QRY_A19) .

ADR_A19 message: ADT patient query result message. This message is the response to
a QRY_A19 patient query message. For details, see 7.3 ADT Query Reply Message
(ADR_A19).

2 Network communication protocol layer structure
The Biolight PDS protocol is defined above the TCP / IP layer, and the figure below shows
the communication layer involved in the communication between the Biolight monitor and the
client PC. The blue layer is defined in this document.
2.1 HL7 Low-Level Protocol
TCP/IP is a byte-stream protocol that does not provide message boundaries. HL7 , which
belongs to application layer protocol, defines message format but does not provide any mechanism
to detect message start and end. To mark the boundaries, the Minimal Lower Layer Protocol is
used. On the network, a message is transmitted in the following format:

Message start mark (1 byte) that corresponds to ASCII <VT>. Its hexadecimal value is
<0x0B>.

HL7 message of multiple bytes.
1．It includes only ISO 8859-1 characters (hexadecimal values between 0x20 and
0xFF) and <CR>. must not contain other control/non-printable characters.
2．When a Biolight product receives a message containing illegal control characters, no further
processing is performed.


Message end mark (2 bytes) that respectively corresponds to ASCII <FS> and <CR>.
Their respective hexadecimal values are <0x1C> and <0x0D>.
2.2 HL7 Protocol Layer
The HL7 protocol is a communication protocol developed by the American HL7 company
for data exchange between medical systems. This protocol has been widely used in many
countries and regions in the world.
Biolight PDS protocol is based on HL7 protocol version 2.6. It uses only part of it.
2.3 Character Set
Biolight PDS protocol supports multiple character sets. The default character set is UTF-8.
Character sets are defined in segment MSH.
MSH|^~\&|||||||ORU^R01|154155500|P|2.3.1||||||UTF-8|
The ISO8859_1 character set can be omitted if it is default.

4 Unsolicited Send Interface
This section introduces the specific content and implementation of unsolicited send interface,
mainly from two aspects of communication process and data communication.
4.1 Communication Process
4.1.1 Network Connection

Active sending interface as server:
As the server of TCP communication, the active sending interface always listens for
connection requests from TCP clients. Once the number of clients requesting connection reaches
the maximum limit, the active sending interface rejects new connection requests. After the
connection is successful, the active sending interface, as a server, does not disconnect from the
client. Interruption and disconnection can only be initiated by the client. In the event of an
unexpected network outage, which requires the client to re-establish the connection, the data
generated during the outage will also be lost. After the connection is reconnected, the active
sending interface sends the data generated after the connection. In addition, it should be noted that
the active sending interface is only responsible for processing the connection request,
disconnection request and sending data to the client, and does not receive other requests in
addition to the request, such as query requests.
If a physical network connection is interrupted during the connection, the Windows TCP
SOCKET may fail to notify the active sending interface, leading us to believe that the socket is
still connected properly. However, because the physical network connection is down, data cannot
be sent or received properly. Therefore, if the client does not receive any data sent by the active
sending interface within the set interval, the client is advised to resend the connection request. If
the connection still fails to be established, take measures to rectify the network fault.

Active sending interface as client:
Active Sending When the interface functions as the TCP communication client, it sends a
connection request to the server every 1s until the connection is successful. After the connection is
successful, the active sending interface, as the client, does not disconnect from the server. If an
unexpected network interruption occurs, the active sending interface sends a connection request to
the server again one second later. If the connection fails, the active sending interface tries to
connect to the server once every one second until the connection is successful.
4.1.2 Network connection Settings

Active sending interface as server: the client needs to know the IP address and port
number of the server to connect. The server IP address can be viewed by clicking the
[Main menu] button on the monitor and [System Info]. The port number is fixed as
6201.

4.1.3 Sending process
After the TCP connection is established, the active sending interface sends the first
ORU^R01 message to the other end through the socket, and then sends the latest monitor data
(ORU^R01 message) at a set interval. The sent data and interval can be set on the HL7 setup
interface, see 4.1.4 for details.
When the other end receives the data, it can send an ACK or NACK in response, but the
active sending interface does not process this response message.
Aperiodic parameters such as NIBP and C.O are also sent at the set interval, and the sent
value is the last manually measured value.

waveform data will be sent.

interface to set, click [Send parameter interval] to set a reasonable sending interval.

4.2 ORU_R01 Message
The ORU in ORU_R01 is the unsolicited observation message, R01 is the event to be
triggered, and the type of the message sent by the unsolicited send interface is the ORU
unsolicited observation message. An HL7 message contains data for one patient.
The ORU_R01 message is divided into five parts: message header, software segment, user
authentication, patient result, and continuous pointer.
The patient results are divided into two parts: the patient and the orderly observation.The
patient part mainly includes patient identification (PID), additional demographic information
(PD1), description and annotation (NTE), observations (OBX), visit (patient visit (PV1) and visit
additional information (PV2)); Order observation mainly includes general order (ORC),
observation report ID (OBR), description and annotation (NTE), observation role (ROL), timing /
quantity (timing / quantity (TQ1), order sequence (TQ2)), contact data (CTD), observation
(observation / result (OBX), description and annotation (NTE)), financial affairs (FT1), clinical
trial identification (CT1), sample (sample (SPM), sample involved result (OBX)).
The sections used this time are:

MSH _ 1 (field separator), MSH_2 (code character), MSH_9 (message type), MSH_10
(message control ID), MSH_11 (process ID), MSH_12 (version ID) for the message
header (MSH);

PID_3 (list of patient identifiers), PID_5 (patient name), PID_7 (date/time of birth),
PID_8 (sex) of patient identification(PID);

PV1_2 (patient class), PV1_3 (designated patient location), and PV1_7 (receiving
doctor) of patient treatment (PV1);

OBR_4 (set ID-OBR), OBR_7 (observation date / time) of observation ID (OBR);

OBX_2 (valuetype), OBX_3 (observation identifier), OBX_4 (observed sub-ID),
OBX_5 (observation results), OBX_11 (observation results status) of the observation
results (OBX);

4.3 Waveform Data Sending
4.3.1 Message content
The waveform message consists of four parts: waveform channel definition data (CHN),
timing information (time and date observed), actual waveform data (WAV) and waveform
annotation (ANO). The waveform data of each module can be either single channel or
multi-channel, and the data of each channel can use separate timing information, channel
definition data, digital waveform data and waveform annotation. In addition, depending on the
situation, the waveform data channels of multiple sensors can share timing information, channel
definition data, and waveform annotations.

5 Query Request Interface
This section introduces the relevant content of the query interface.
5.1 Communication process
5.1.1 Network Connection
The query interface is only used as the server to wait for connection requests from the client,
and the process of establishing a connection is the same as actively sending the interface as the
server. Therefore, please refer to section 4.1.1.
5.1.2 Network Connection Settings
Query interface as server: The client needs to know the IP and port number of the server in order
to connect. The IP address of the server can be viewed by clicking on the [Main menu] button of
the monitor and clicking on the [System Info]. The port number is fixed to 6202.
5.1.3 Data Interaction
The query interface adopts a communication mode of request and response, which means that
the server will not send any data to the client until it receives the query message (QRY_R02) sent
by the client.
The client can query the data of one, multiple, or all monitors at once, and can specify the
type of data to be queried. As the query interface of the server, after receiving the query message,
it will first return an ACK message as a response, and finally return a message containing the
query result (ORF_R04). After receiving the result returned by the interface, the client will send
an ACK or NCK as a response, but this response will be ignored. If the server does not specify the
data type to query, the interface will not return any messages.
If the client sends a query message followed by a new query message, the interface will first
respond to the previous query message, and then send an ACK message warning the client that the
query message interval is too short. Therefore, when sending query messages, it is important for
the client to keep track of the time interval and wait at least 10 seconds before sending new query
messages.
5.1.4 Send Content
The following data is sent after the client sends the correct query message.
1．Module parameter values.
2．Physiological alarm data.


=== PAGE 38 ===

3．Technical alarm data.
4．Module configuration and device status include: device standby state, physiological and
technical highest alarm level, device alarm state, ECG lead type, PR source, RR source.
5．Module parameter alarm configuration data, including alarm switch, alarm level, and
alarm upper and lower limit values.
5.2 QRY_R02 Message
QRY_ The QRY in R02 represents the query observation results, and R02 represents the
event. The query message type sent by the client is QRY_ R02 generally forms a query and
response mode with ORF.
QRY_ R02 is divided into three parts, namely Message Header (MSH), Query Definition
(QRD), and Query Filter (QRF).

6 Real time query interface
The difference between real-time interfaces and other interfaces is that only one patient's data
is sent in a TCP connection. The patient's personal information does not appear in every HL7
message (that is, PID and PV1 segments do not appear in each HL7 message), and is only sent
after the connection is established or changed. In addition, this interface also sets up UDP
broadcast information to send machine system information and patient personal information.
6.1 Communication process
6.1.1 Network Connection
The method and process for establishing a connection between the client and server of the
real-time query interface are the same as those for actively sending the interface, so we can refer
to 4.1.1.
6.1.2 Network Connection Settings
Real time query interface as the server: The client needs to know the IP and port number of
the server in order to connect. The IP address of the server can be viewed by clicking on the [Main
menu] button of the monitor and clicking on the [System Info]. The port number is fixed to
6210. In addition, the real-time query interface is also set with UDP broadcast information, which
will send broadcast information to all hosts in the subnet where the monitor is located, with port
number 6234.
6.1.3 Data Interaction
The real-time query interface adopts a communication mode that combines request and
response with active sending. That is, after the client establishes a connection with the interface,
the interface first sends online module information to the client, and then sends heartbeat packets
every second. At the same time, the client needs to send a heartbeat packet to the interface. If the
interface does not receive the heartbeat packet sent by the client within 10 seconds after
establishing the connection, it will actively disconnect.
During the process of maintaining network connectivity, the client can send a message that
needs to be queried (QRY_R02) and specify a response time interval. After receiving the correct
query message, the interface will reply to the corresponding (ORU_R01) result message based on
the time interval.
If the client sends a query and message followed by a new and different query message, the
interface returns the result message of the new query message first, and then the previous result
message. Each subsequent result message will also be sent in this way.

The real-time query interface also sets up UDP broadcast information, sending machine
system information and patient personal information every second to inform other hosts on the
network of its existence.
6.1.4 Send Content
6．Online broadcast information: including patient name, gender, doctor ID, date of birth,
department, bed number, and monitor name;
7．TCP heartbeat packet information: sent once every 1 second;
8．Patient information: When the TCP connection is successfully established, the latest
patient information will be sent to the client every second;
9．Response message: After receiving the query message, the interface will return the
corresponding response message;
10．Physiological parameters:
When the interface receives
a query message
as a
physiological parameter, it sends physiological parameters according to the time interval;
11．Physiological alarm information: When the interface receives a query message as a
physiological alarm, it sends physiological alarm information according to the time interval;
12．Technical alarm information: When the interface receives a query message as a technical
alarm, it sends technical alarm information according to the time interval;
13．Module online and offline information: After establishing a TCP connection between the
interface and the client, the interface will send the online module information once, and then send
it again if there are any changes afterwards.
14．Parameter alarm configuration information: When the interface receives a query
message as a technical alarm, it sends parameter alarm configuration information according to the
time interval;

7 ADT Net Query Interface
This section introduces the content of the ADT query interface, which mainly involves the
communication process between the ADT query interface and the BltADTServer server of
Biolight development.
7.1 Communication Process
7.1.1 Network Connection
This interface is the client that establishes the TCP connection, and BltADTServer is the
server that listens.
After a TCP connection is set, the TCP interface does not actively close the connection. If the
TCP connection is interrupted or closed, the interface automatically reconnects to the TCP within
a fixed period of time.
All ADT messages from both sides are transmitted in the same TCP connection. The
following two sections detail the data communication process in both transmission directions.
7.1.2 Data Interaction
This interface sends a query message in QRY^A19 format. The QRY^A19 message contains
a patient medical record number (MRN) as a unique query condition.
The BltADTServer server executes the exact query and returns an ADR^A19 message based
on the query criteria. If patient information is found that matches the query criteria, patient records
that meet the query criteria are returned. If no patient information is found that matches the query
criteria, the message returned does not contain any patient information. If the HIS/CIS system
does not return any message within a limited period of time, the interface treats it as a query
timeout.
After receipt of the HL7 message, the interface returns an ACK message indicating whether
the format is correct. This interface does not perform any operation when receiving an ACK
message.
